ПРАВИЛА ЧТЕНИЯ ЭТОГО ДОКУМЕНТА
--------------------------------------------------------------------------------
1. Каждое значение либо ТОЧНО измерено по скриншоту (помечено "Источник: скриншот"),
   либо является оценкой (помечено "Estimated Value") с указанием уровня уверенности:
   Very High / High / Medium / Low.
2. Все оценочные значения подчиняются единой сетке отступов (8pt grid, база 4px) и
   единой типографической шкале — см. Раздел 3 и Раздел 5.
3. Единицы измерения — CSS px (логические точки, 1x). Реальные экраны iPhone рендерят
   это с pixel ratio 2x или 3x (Retina), см. Раздел 1.
4. Документ построен так, чтобы его можно было скормить целиком в контекст LLM-агента
   (Claude Code) и получить воспроизведение интерфейса без дополнительных уточнений.
5. Порядок разделов соответствует индустриальному стандарту handoff-документации
   (аналог Figma Dev Mode + Design Tokens Spec + Storybook Docs).

================================================================================
СЕКЦИЯ 1 — ИНФОРМАЦИЯ О ПРИЛОЖЕНИИ
================================================================================

1.1 Тип приложения
    Personal Finance Management (PFM) — трекер личных финансов / бюджета.
    Экраны: Главный дашборд баланса, модальная форма добавления операции,
    полный список операций с поиском и фильтрами.
    Язык интерфейса: русский (ru-RU). Формат валюты: RUB, символ "₽",
    формат чисел с разделителем тысяч (пробел) и двумя знаками после запятой,
    запятая как десятичный разделитель: "4 452,00 ₽".
    Формат даты: "13 июля 2026 г." (день, полное название месяца в родительном
    падеже, год, сокращение "г."). Формат времени: 24-часовой "13:40".

1.2 Платформа
    Progressive Web App (PWA), открываемый через Safari на iOS.
    Признаки: нативная iOS статус-бар с временем "9:41" (Apple demo-time стандарт
    для скриншотов), иконки сигнала/Wi-Fi/батареи в стиле iOS 17/18, отсутствие
    браузерного chrome (адресной строки) на кадрах — предполагается, что скриншоты
    сделаны либо в режиме "Add to Home Screen" (standalone display-mode), либо
    выше линии обрезки браузера намеренно исключена. Рекомендуется верстать под
    display-mode: standalone с manifest.json.

1.3 Операционная система
    iOS (версия 17–18, судя по стилю иконок статус-бара и отсутствию "notch"-выемки
    старого типа). Скругления экрана — программные, не аппаратные (веб-контент).

1.4 Рендеринг-окружение
    WebView / Safari mobile. Рекомендуемый стек воссоздания: React + Tailwind CSS
    (см. системный промпт пользователя — упоминание Figma Auto Layout транслируется
    в CSS Flexbox с gap). Все скругления, тени и градиенты должны быть реализованы
    через CSS (border-radius, box-shadow, linear-gradient / radial-gradient),
    шрифты — через system font stack с фолбэком на Inter для кириллицы.

1.5 Safari PWA — специфика
    - viewport meta: width=device-width, initial-scale=1, viewport-fit=cover
    - Учитывать env(safe-area-inset-top) и env(safe-area-inset-bottom) — критично,
      т.к. нижняя навигация и верхний статус-бар должны учитывать вырез/индикатор.
    - Status Bar Style: предположительно "default" (тёмные иконки на светлом фоне),
      т.к. фон приложения светлый (лавандово-розовый градиент).
    - Home Indicator (нижняя полоса свайпа iOS): не видна на скриншотах явно, но
      должна учитываться как safe-area-inset-bottom ≈ 34px (iPhone с Face ID).

1.6 Предположения об устройстве
    Референсное устройство: iPhone 14/15/16 (без Pro Max), стандартный размер.
    Экран: 390 x 844 логических точек (pt) — наиболее вероятный базовый макет,
    судя по пропорциям кадров (соотношение сторон скриншотов ≈ 0.435, что близко
    к 390/844 = 0.462, но с учётом того что скриншоты обрезаны без строки браузера
    снизу — итоговая оценка).
    Estimated Value | Confidence: High

1.7 Экранное разрешение
    Логические точки (points): 390 x 844 pt (базовый макет дизайна).
    Pixel Ratio: 3x (Retina, iPhone стандарт) → физический рендер 1170 x 2532 px.
    Estimated Value | Confidence: High

1.8 Safe Areas (относительно 390pt ширины)
    - Top safe area (status bar): 44pt (стандарт iPhone с Face ID / Dynamic Island
      без острова, обычный статус-бар)
    - Bottom safe area (home indicator): 34pt
    - Left/Right safe area: 0pt (нет выемок по бокам в портретной ориентации)
    Estimated Value | Confidence: High

1.9 Dynamic Island
    Не обнаружено визуальных признаков Dynamic Island на скриншотах (верхняя часть
    экрана — обычный статус-бар без выреза-пилюли). Приложение либо тестировалось
    на устройстве без Dynamic Island (iPhone SE/13/14 стандарт), либо скриншот
    обрезан выше выемки. Рекомендация для верстки: не завязываться жёстко на
    Dynamic Island, использовать стандартный safe-area-inset-top = 44–59pt
    (гибкая переменная).
    Estimated Value | Confidence: Medium

СЕКЦИЯ 2 — CANVAS (ХОЛСТ)
================================================================================

2.1 Canvas Width
    390px (логическая ширина экрана, базовый артборд Figma "iPhone 14/15").
    Источник: пропорции скриншота. Confidence: High

2.2 Canvas Height
    844px — полный артборд под iPhone 14/15 (стандарт, не Pro Max, не mini).
    Реальные PNG-скриншоты имеют высоту, кратную этому базовому размеру с учётом
    масштабирования экспорта.
    Estimated Value | Confidence: High

2.3 Safe Area (используемая зона контента)
    Top: 44px (под статус-бар, контент начинается ниже)
    Bottom: 34px (над home indicator)
    Эффективная зона контента: 390 x 766px (844 - 44 - 34 = 766)

2.4 Margins (внешние отступы экрана)
    Горизонтальный margin (left/right): 20px на каждую сторону — единый на всех
    трёх экранах (совпадает у карточки баланса, списка операций, шапки).
    Источник: измерено по позиции левого края карточек/текста относительно
    края экрана на всех 3 скриншотах — консистентно 20px.
    Confidence: High

2.5 Outer Padding
    Top padding контентной области (от низа статус-бара до первого элемента,
    напр. "Добрый день,"): 16–20px.
    Estimated Value | Confidence: Medium-High
    Bottom padding перед нижним таб-баром: контент скроллится "под" таб-бар,
    таб-бар — floating/fixed элемент с собственным фоном (см. 2.7).

2.6 Scrollable Area
    Вся область контента между шапкой (не залипает) и нижним таб-баром является
    вертикально скроллируемой (overflow-y: auto / -webkit-overflow-scrolling: touch).
    На Экране 1 (Главная): скроллится весь столбец от приветствия до конца списка
    операций.
    На Экране 3 (Все операции): скроллится область под строкой поиска/фильтрами
    (сама строка поиска и фильтр-чипы предположительно НЕ sticky на основе кадра,
    но рекомендуется сделать их sticky top для UX — см. примечание в Секции 13).

2.7 Visible Viewport / Fixed elements
    - Верхний статус-бар: нативный iOS, не часть DOM-верстки, но должен быть учтён
      отступом.
    - Нижний Tab Bar: FIXED / зафиксирован внизу экрана (position: fixed или
      sticky bottom), белый фон, скруглённые верхние углы, поверх скроллящегося
      контента с overlay-эффектом (контент уходит под таб-бар с небольшим fade,
      либо просто обрезается — см. кадр 1, где последняя карточка "Подарки"
      подрезана таб-баром снизу — подтверждает, что Tab Bar — floating поверх
      контента с собственной тенью/фоном).
      Источник: скриншот 1 (нижняя карточка операции обрезана баром).

2.8 Maximum Content Width
    390px (mobile-only дизайн, без десктопной адаптации в рамках этих кадров).
    Все карточки растягиваются на 100% доступной ширины минус горизонтальные
    margins (390 - 40 = 350px полезной ширины).

2.9 Content Alignment
    Вертикальный стек (single column), выравнивание по левому краю для текстовых
    блоков, элементы карточек — flex row с justify-content: space-between там, где
    есть значение справа (суммы, шевроны).

СЕКЦИЯ 3 — GRID SYSTEM (СЕТКА)
================================================================================

3.1 Базовая единица отступа (Base Spacing Unit)
    4px — базовая единица. Все отступы кратны 4, преимущественно кратны 8
    (стандартный 8pt grid с использованием 4px для "полушага" в мелких элементах
    типа gap между иконкой и текстом).
    Confidence: High (типографический и отступный ритм по всем 3 кадрам консистентен)

3.2 Шкала отступов (Spacing Scale) — токены
    space-1  = 2px   (микро-корректировки, редко)
    space-2  = 4px   (gap между строками мелкого текста)
    space-3  = 8px   (gap между иконкой суммы и текстом; внутренние мелкие отступы)
    space-4  = 12px  (gap между иконкой транзакции и текстовым блоком)
    space-5  = 16px  (стандартный gap между элементами списка; padding кнопок)
    space-6  = 20px  (горизонтальный margin экрана; padding карточек)
    space-7  = 24px  (padding крупных карточек — Balance Card; radius крупных карт)
    space-8  = 32px  (вертикальные разрывы между крупными секциями)
    space-9  = 40px  (редкие крупные разрывы, напр. над Tab Bar)
    space-10 = 48px  (высота маленьких иконок-плашек категорий в некоторых состояниях)

3.3 Вертикальный ритм (Vertical Rhythm)
    Между крупными блок-секциями (Header → Balance Card → CTA → Insight →
    Section List): 16px между карточками, 24–32px после шапки приветствия.
    Между элементами списка транзакций: разделительная линия (divider) 1px,
    вертикальный padding элемента списка сверху/снизу ~14–16px (итоговая высота
    строки транзакции ≈ 64–72px, см. Секцию 9, компонент Transaction List Item).

3.4 Горизонтальный ритм (Horizontal Rhythm)
    Все карточки: одинаковый горизонтальный margin 20px от края экрана.
    Внутри карточек: горизонтальный padding 20px (Balance Card, Insight Card),
    16px (мелкие суб-карточки Доходы/Расходы внутри Balance Card).

3.5 8pt Grid — подтверждение
    Высоты интерактивных элементов кратны 8: кнопка "+ Новая операция" ≈ 56px,
    Save-кнопка ≈ 56px, иконки-плашки категорий 56x56 / 64x64, аватар/bell-icon
    44x44 (кратно 4, минимальный touch target Apple — см. Секцию 14).

3.6 4pt Grid
    Используется для тонкой настройки: line-height текста, gap 4px между
    заголовком и подзаголовком строки транзакции, borders 1px/1.5px.

3.7 Column Assumptions
    Это НЕ многоколоночная сетка в классическом вебовском смысле (нет 12-колоночной
    сетки) — мобильный single-column layout. Внутренние grid-структуры:
    - Категории (Экран 2): горизонтальный ряд из 5 элементов равной ширины
      (auto-layout row, space-between/gap фиксированный, возможна горизонтальная
      прокрутка если категорий больше 5).
    - Доходы/Расходы (Экран 1): 2-колоночная сетка 50/50 с gap 12px.

3.8 Container Width
    Полезная ширина контента: 350px (390 - 2*20).
    Внутри Balance Card: 350 - 2*20(padding) = 310px полезной внутренней ширины.

3.9 Полная система отступов — сводная таблица
    ┌─────────────┬──────┬─────────────────────────────────────────────┐
    │ Token       │ px   │ Применение                                   │
    ├─────────────┼──────┼─────────────────────────────────────────────┤
    │ space-1     │ 2    │ микро-выравнивание badge/индикаторов         │
    │ space-2     │ 4    │ gap заголовок/подзаголовок в списке           │
    │ space-3     │ 8    │ gap иконка+текст (мелкие), padding badge      │
    │ space-4     │ 12   │ gap иконка транзакции ↔ текстовый блок        │
    │ space-5     │ 16   │ gap между карточками секций, padding кнопок   │
    │ space-6     │ 20   │ margin экрана, padding карточек               │
    │ space-7     │ 24   │ padding Balance Card, radius крупных карт     │
    │ space-8     │ 32   │ разрыв между макро-секциями                   │
    │ space-9     │ 40   │ отступ над Tab Bar                            │
    │ space-10    │ 48   │ доп. крупные иконки                           │
    └─────────────┴──────┴─────────────────────────────────────────────┘

    4.5.1 ПРИМЕЧАНИЕ О НЕСОГЛАСОВАННОСТИ (обязательно зафиксировать для инженера):
    Иконка "Зарплата" на Экране 1 (Главная, список "Последние операции") имеет
    ГОЛУБОЙ фон (#E3EEFB), тогда как на Экране 3 (Все операции) та же категория
    "Зарплата" отображается с ЗЕЛЁНЫМ фоном (#DFF3EA/#D8F5E3). Это расхождение
    между кадрами — либо (а) ошибка дизайна в исходном макете, либо (б) иконка
    цветокодируется не по категории, а по ТИПУ операции (доход = зелёный тон,
    вне зависимости от категории), а на Экране 1 применена категориальная
    (не типовая) окраска как исключение/баг.
    РЕКОМЕНДАЦИЯ ДЛЯ ВОССОЗДАНИЯ: реализовать окраску иконки-плашки ПО ТИПУ
    ОПЕРАЦИИ (доход → зелёная гамма, расход → цвет категории, перевод → синяя
    гамма), т.к. это более логичная и повторяемая система, соответствующая

    случаем/исключением исходных данных, а не паттерном дизайн-системы.
    Confidence: Medium (интерпретация неоднозначного визуального свидетельства)

4.6 GLASS EFFECTS / BLUR / OPACITY

    rgba(255,255,255,0.96)) поверх сплошного не блюрящегося фона.
    Тем не менее для аутентичного iOS-ощущения РЕКОМЕНДУЕТСЯ (опционально,

        --glass-blur: blur(20px)

    т.к. под ним видно, что нижняя карточка транзакции слегка обрезается —
    характерное поведение iOS blur-материала.
    --tabbar-bg: rgba(255,255,255,0.92)
    --tabbar-blur: blur(24px)
    Confidence: Medium (основано на паттерне iOS HIG, не на прямом визуальном

    400 (Regular) — тело текста, подписи, плейсхолдеры
    500 (Medium)  — лейблы полей, табы навигации
    600 (SemiBold) — акцентный текст, названия транзакций, кнопки
    700 (Bold)     — заголовки, суммы, крупные цифры

5.5 Line Height Rule
    Общее правило: line-height = font-size * 1.25–1.3 для всех body/caption
    стилей; для display/heading — 1.18–1.2 (плотнее, характерно для крупных
    цифр баланса).

5.6 Paragraph Spacing
    Между заголовком строки и подзаголовком (напр. "Еда" / "Покупки и кафе"):
    margin-top: 2px (space-2 сокращённый до 2px для очень плотной пары строк).
    Между лейблом поля и полем ввода (напр. "Категория" → ряд иконок):
    margin-bottom: 12px (space-4).

5.7 Text Alignment
    Весь основной текст — left-aligned. Исключение: суммы в списке транзакций
    и значения (right-aligned в своей колонке flex-строки), кнопки — центр
    (text-align: center внутри pill/button).

5.8 Числовой формат (важно для инженера)
    Разделитель тысяч — узкий неразрывный пробел " " (U+202F или обычный
    пробел), десятичный разделитель — запятая, всегда 2 знака после запятой,
    символ валюты "₽" после числа через обычный пробел.
    Пример: "12 000,00 ₽", "-2 000,00 ₽", "4 452,00 ₽".
    Знак "+" явно показывается перед положительными суммами дохода
    ("+12 000,00 ₽"), знак "-" перед расходами.

8.3 Padding иконок
    Внутри цветных rounded-square/circle подложек иконка/эмодзи центрируется с
    равным padding со всех сторон: (контейнер - иконка) / 2.
    Пример: контейнер 48px, эмодзи 24px → padding 12px со всех сторон.

8.4 Правила цвета иконок
    - Функциональные UI-иконки (bell, back, filter, search, calendar, close,
      chevron): цвет --color-text-primary (#16192B) для основных, или
      --color-text-secondary для второстепенных/неактивных (напр. шеврон
      справа в строке "Дата" — серый).
    - Активные/акцентные иконки (Tab Bar активный таб, "+" в кнопке): цвет
      --color-primary-500 (#7C6EF2).
    - Иконки категорий: полноцветные (эмодзи) либо в цвете --catX-icon token.
    - Стрелка "Доходы": --color-success-600, стрелка "Расходы": --color-error-600.

8.5 Alignment
    Иконка + текст всегда выровнены по vertical-center внутри своей строки
    (align-items: center во flex-row).

СЕКЦИЯ 13 — ВЗАИМОДЕЙСТВИЕ (INTERACTION)

13.1 Навигация верхнего уровня
    BottomTabBar управляет 4 основными разделами: Главная, Дашборд, Категории,
    Аккаунт. Переключение таба — мгновенная смена контента (client-side route
    change), без полноэкранного transition-анимации (типично instant swap
    с возможной лёгкой fade-анимацией контента, 150-200ms ease-out).
    "Все операции" (Экран 3) технически является под-маршрутом раздела
    "Дашборд" (подтверждено активным табом "Дашборд" на этом экране), но
    визуально имеет собственный ScreenHeaderNav с BackButton — т.е. это
    навигация вглубь стека (push), а не смена таба.

13.2 Transition: Home → AllTransactions
    Триггеры: тап по "Смотреть все" (SectionHeader на Главной) ИЛИ переход
    через таб "Дашборд" в нижней навигации.
    Анимация: horizontal slide-in справа налево (push navigation, стандарт
    iOS), длительность ≈300ms, easing cubic-bezier(0.32, 0.72, 0, 1)
    (характерная iOS-кривая для push-переходов).
    Обратно: BackButton (стрелка влево) в ScreenHeaderNav → slide-out обратно,
    либо системный свайп-жест от левого края экрана (edge swipe back).

13.3 Transition: Home → NewTransactionSheet
    Триггер: тап по PrimaryPillButton "+ Новая операция" (Экран 1) ИЛИ по
    аналогичной FAB/кнопке на других экранах (Категории/Аккаунт, гипотетически).
    Анимация: vertical slide-up (bottom sheet presentation), длительность
    ≈350-400ms, easing ease-out, начинается от нижнего края экрана и
    поднимается до текущей высоты (см. 10.2.1 — почти полный экран).
    Фон предыдущего экрана: затемняется overlay --opacity-overlay (0.4) ЛИБО
    (что более вероятно судя по единству фона) полностью подменяется fullscreen
    modal без видимого предыдущего контента.
    Закрытие: тап по CloseButton (X) ИЛИ свайп вниз по DragHandle/всей area
    (drag-to-dismiss, стандартный iOS bottom sheet жест) ИЛИ тап на area
    выше модалки (если есть видимый scrim).

13.4 Кнопка "Сохранить" — состояния и поведение
    - disabled state: пока "Сумма" = 0,00 ₽ ИЛИ не выбрана категория —
      кнопка неактивна (opacity 0.4, non-interactive). Estimated поведение,
      Confidence: Medium (стандартная UX-практика форм, не показано впрямую
      на статичном кадре, где сумма = 0,00, а категория уже выбрана —
      возможно кнопка активна сразу при выборе категории независимо от суммы,
      либо активна всегда и валидация происходит при сабмите с inline-error).
    - При успешном сохранении: закрытие sheet + вставка новой строки в
      TransactionListItem на Главной с лёгкой highlight-анимацией (flash
      background --color-primary-50 → transparent за 600ms) — рекомендация
      для приятного фидбека, не подтверждено кадром.

13.5 TransactionTypeSegmentedControl — переключение
    Тап по неактивному сегменту → анимированный сдвиг активного "плашки"-
    background (background pill) к новой позиции: transform translateX +
    изменение цвета фона/текста, длительность ≈200ms ease-in-out.
    При переключении на "Доход" или "Перевод" — набор доступных категорий
    в CategorySelectorSection должен соответственно измениться (напр., для
    "Доход" — категории "Зарплата", "Подарки", "Перевод", "Другое"; для
    "Перевод" — список счетов вместо категорий). Это не показано на кадрах
    напрямую (виден только "Расход"), но логически необходимо для полноты
    системы. Confidence: Low-Medium (продуктовая гипотеза).

13.6 Скролл-поведение
    Home (Экран 1): весь контент скроллится вертикально под фиксированным
        TabBar; Header (приветствие) НЕ sticky, уходит вверх при скролле.
    AllTransactions (Экран 3): РЕКОМЕНДУЕТСЯ сделать SearchBar +
        FilterChipGroup sticky top (position: sticky; top: {высота
        ScreenHeaderNav}) при скролле длинного списка транзакций — общепринятый
        UX-паттерн для списков с поиском/фильтрами. Сам ScreenHeaderNav
        (заголовок "Все операции") также может стать компактным sticky-баром
        при скролле (collapsing large title, iOS-паттерн), но это опциональное
        улучшение сверх строгого воссоздания кадра.
    NewTransactionSheet: внутренняя область формы скроллится при нехватке
        высоты экрана, кнопка "Сохранить" остаётся видимой (sticky bottom
        внутри модалки — см. примечание 10.2.2).

13.7 Жесты (Gesture assumptions)
    - Свайп вниз по BottomSheetModal → закрытие (drag-to-dismiss)
    - Свайп от левого края экрана вправо → back-navigation (edge swipe,
      системный iOS-жест, работает на Экране 3 → Экран 1)
    - Долгое нажатие на TransactionListItem (гипотетически) → контекстное
      меню (редактировать/удалить), не подтверждено кадрами, но типично для
      подобных приложений. Confidence: Low.
    - Тап на строку транзакции → переход на детальный экран транзакции
      (не показан в предоставленных кадрах). Confidence: Low-Medium
      (стандартный паттерн, но не подтверждён).
    - Тап на "Баланс" eye-icon → toggle видимости суммы (маскирование
      "4 452,00 ₽" → "•••• ₽" для приватности при просмотре в публичном месте).

13.8 Состояния кнопок и интерактивных элементов — общие правила
    Все тап-элементы (кнопки, чипы, иконки, строки списка):
    - pressed: opacity 0.85 ИЛИ scale(0.97), длительность transition 100ms
    - focus (для клавиатурной/VoiceOver навигации, актуально для веб-версии
      PWA): outline 2px solid --color-primary-500, offset 2px
    - hover (нерелевантно для мобильного тача, но должно быть предусмотрено
      если PWA открывается на iPad/десктопе с трекпадом): лёгкое затемнение
      background на 4-6%

13.9 Ввод данных
    - AmountInputBlock: числовая клавиатура (inputmode="decimal"), маска
      формата "0,00 ₽" с автоматическим сдвигом разрядов при вводе цифр
      справа налево (банковский паттерн ввода копеек).
    - Комментарий: обычная текстовая клавиатура, single-line с возможностью
      auto-grow до 2-3 строк при длинном тексте.
    - Дата: тап по NavigationField "Дата" открывает нативный/кастомный
      date-picker (bottom sheet поверх текущей формы либо inline calendar
      picker), значение по умолчанию — текущая дата ("13 июля 2026 г." —
      совпадает с "сегодня" в контексте кадров).
    - Теги: тап открывает отдельный экран/sheet со списком существующих
      тегов + возможностью создать новый (multi-select chips).

13.10 Поиск (Экран 3)
    Тап по SearchBar → фокус на инпуте, появление курсора, возможно
    сворачивание FilterChipGroup или появление клавиатуры снизу с сохранением
    видимости списка результатов в реальном времени (live filter по мере
    ввода, debounce ≈200-300ms).

13.11 Фильтр-чипы (Экран 3)
    Тап по "Расходы"/"Доходы" → переключение активного состояния (только один
    активен одновременно, как radio-behaviour: "Все"/"Расходы"/"Доходы" —
    взаимоисключающие), список ниже мгновенно перефильтровывается без
    отдельного экрана загрузки.
    Тап по CalendarIconButton → открытие date-range picker (bottom sheet)
    для фильтрации по произвольному диапазону дат.
    Тап по FilterIconButton (в ScreenHeaderNav) → открытие расширенного
    bottom sheet с доп. фильтрами (по категории, по сумме, по тегам) —
    гипотетический функционал, не показан в кадрах. Confidence: Low.

СЕКЦИЯ 14 — ДОСТУПНОСТЬ (ACCESSIBILITY)

14.1 Touch Targets
    Apple HIG минимум: 44x44pt для любого интерактивного элемента.
    Проверка по компонентам:
    - NotificationBell: 44x44 ✓ (соответствует)
    - BackButton: визуальная иконка 24x24, но touch target должен быть
      расширен паддингом до 44x44 ✓ (обязательно предусмotreть invisible
      hit-area больше видимой иконки)
    - CloseButton (X): аналогично, 44x44 hit-area при визуальной иконке 24x24
    - TabBarItem: ширина ≈97.5px, высота ≥48px — превышает минимум ✓
    - CategoryTile: 64px width — превышает минимум по горизонтали ✓, но
      высота IconTile сама по себе 64px — ✓ в норме
    - FilterChip: height 40px — НИЖЕ рекомендованного минимума 44px.
      РЕКОМЕНДАЦИЯ: увеличить hit-area невидимым padding до 44px по высоте,
      сохранив визуальную высоту чипа 40px (padding trick).
    - CalendarIconButton/FilterIconButton: 40x40 — аналогично, расширить
      hit-area.

14.2 Контраст (WCAG)
    --color-text-primary (#16192B) на --color-surface-card (#FFFFFF):
      контраст ≈ 15.8:1 — превышает AAA (7:1) ✓
    --color-text-secondary (#7B7F94) на белом фоне:
      контраст ≈ 4.6:1 — соответствует AA для обычного текста (4.5:1) ✓,
      но НЕ соответствует AAA (7:1) — приемлемо для body text по AA.
    --color-text-tertiary (#A0A3B1, плейсхолдеры) на белом:
      контраст ≈ 2.9:1 — НЕ соответствует даже AA для обычного текста.
      Плейсхолдеры традиционно освобождены от строгих WCAG-требований
      (не являются семантическим контентом), но РЕКОМЕНДАЦИЯ: если возможно,
      затемнить до ≈#8B8E9E для пограничного улучшения читаемости при
      сохранении визуальной "приглушённости".
    --color-primary-500 (#7C6EF2, текст ссылок) на белом:
      контраст ≈ 3.9:1 — НЕ соответствует AA для обычного текста (нужно
      4.5:1), но соответствует AA для крупного текста (3:1, ≥18px/24px bold
      или ≥24px). Т.к. "Смотреть все"/"+ Новая операция" используют 15-16px —
      ПОГРАНИЧНОЕ несоответствие. РЕКОМЕНДАЦИЯ: либо утемнить primary до
      #6A5AE0 (~4.6:1) для текстовых применений, либо считать это
      декоративным/навигационным элементом с сопровождающей иконкой (снижает
      строгость требования).
    Success/Error text (#22C55E / #EF4444) на белом: контраст ≈2.5-3:1 —
      НЕ соответствует AA для обычного текста. Это типичная проблема ярких
      "приятных" зелёных/красных в финансовых приложениях. РЕКОМЕНДАЦИЯ для
      реальной доступности (не обязательно для точного визуального
      соответствия скриншоту): продублировать семантику знаком "+/-"
      (что уже сделано) и не полагаться только на цвет — это частично
      компенсирует WCAG 1.4.1 (Use of Color).

14.3 Font Scaling (Dynamic Type)
    РЕКОМЕНДАЦИЯ: использовать относительные единицы (rem) вместо px в
    финальной CSS-реализации, с базой 16px = 1rem, чтобы поддержать
    пользовательское увеличение текста через настройки iOS
    (Text Size Adjustment). Все touch targets должны сохранять минимум
    44x44 даже при увеличенном шрифте (использовать min-height/min-width,
    а не fixed height, где возможно).

14.4 Accessibility Spacing
    Между соседними интерактивными элементами — минимум 8px зазора, чтобы
    избежать случайных нажатий (mis-tap). Проверено: gap между FilterChip
    элементами (8-10px) — приемлемо, но пограничный минимум; РЕКОМЕНДАЦИЯ
    не уменьшать этот gap при адаптации.

14.5 VoiceOver assumptions (для реализации ARIA/семантики в PWA)
    - NotificationBell: aria-label="Уведомления, есть непрочитанные"
      (динамически убирать "есть непрочитанные" если badge скрыт)
    - EyeToggle: aria-label="Скрыть баланс" / "Показать баланс" (toggle state)
    - TransactionListItem: role="button" (если кликабелен), aria-label
      формируется как "{Название}, {Подкатегория}, {Сумма}, {Время}"
      например: "Еда, Покупки и кафе, минус 580 рублей, 13:40"
    - PrimaryPillButton "Сохранить": aria-disabled синхронизирован с visual
      disabled state
    - TransactionTypeSegmentedControl: role="tablist" / каждый Segment —
      role="tab" с aria-selected
    - CategoryTile: role="radio" внутри role="radiogroup" (выбор одной
      категории из набора), aria-checked для selected state
    - BottomTabBar items: role="tab", aria-current="page" для активного
    - Все декоративные иконки без семантической нагрузки: aria-hidden="true"
      (напр. WalletIllustration, chevron-иконки при наличии текстового label
      рядом)
    - Формы (Комментарий, Теги, Сумма): связаны через <label for> либо
      aria-labelledby с соответствующими текстовыми лейблами
